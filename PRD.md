# 产品名称（暂定）

**Site2Agent / AgentLayer for Websites / Agent-Ready Web**

------

# 1. 产品概述

## 1.1 产品背景

AI 搜索、网页问答、自动化 Agent、编程 Agent 正在逐步成为网站流量和信息分发的新入口。未来很多网站不再只是“给人浏览”，还需要“给 Agent 稳定读取和查询”。

但现在大多数网站存在几个问题：

- 内容结构是面向人类阅读的，不是面向 Agent 理解的
- 公开页面很多，但没有统一的机器可读入口
- Agent 很难判断哪个页面最权威、哪个信息最新、哪个字段最关键
- 网站没有控制“开放给 Agent 什么，不开放什么”的能力
- 中小站长和小团队做不起完整 API、知识图谱或重型 MCP 体系

因此，需要一款产品，帮助网站快速生成一层 **Agent 可读层**，让网站内容对 AI 更友好，同时保留站长控制权。

------

## 1.2 产品定位

**这不是一个“软件转 CLI”的工具。**
 **这是一款“网站转 Agent 可读层”的工具。**

一句话定位：

**把普通网站的公开内容，转成 Agent 能稳定理解、检索、引用和查询的标准化输出层。**

------

## 1.3 产品边界

### 产品做什么

- 扫描网站公开页面
- 识别页面类型
- 提取关键信息
- 生成 Agent 可读入口文件
- 生成结构化 JSON 数据
- 生成基础查询接口
- 提供 Agent 开放规则控制

### 产品不做什么

- 不控制本地软件
- 不把 GUI 软件包装成 CLI
- 不替代网站原有前端
- 不做复杂企业级工作流引擎
- 不做完整操作型 Agent 平台
- 不做“帮 Agent 直接执行外部系统动作”的控制层

------

# 2. 产品目标

## 2.1 核心目标

让网站从“只能被浏览器访问”升级为“也能被 Agent 正确读取和查询”。

## 2.2 商业目标

做成一个适合个人起步、逐步产品化的工具：

- 可以卖给个人站长
- 可以卖给文档站
- 可以卖给 SaaS 官网
- 可以卖给知识库站点
- 可以先工具收费，再走订阅/托管

## 2.3 关键成功指标

### 使用指标

- 首次接入时间 < 30 分钟
- 非复杂网站可自动完成基础结构化输出
- 站长可手动配置开放范围

### 效果指标

- Agent 对网站关键信息提取准确率明显提升
- AI 回答中引用站点内容时更稳定
- 文档/FAQ/产品信息被检索到的成功率提升

### 商业指标

- 有用户愿意为“让网站更容易被 AI 理解”付费
- 至少一个垂直场景形成模板化交付

------

# 3. 目标用户

## 3.1 用户类型

### A. 个人站长 / 博主

需求：

- 希望博客、教程、导航内容更容易被 AI 检索和引用
- 不想自己写复杂接口
- 想低成本接入

### B. 文档站维护者

需求：

- 希望文档被 Agent 正确读取
- 不希望 AI 读错版本、读错段落
- 希望输出清晰的 docs 索引和搜索接口

### C. 小型 SaaS / 工具站

需求：

- 希望产品信息、价格、功能、FAQ 能被 AI 正确读取
- 希望官网可以作为 AI 客服/销售的公开知识源
- 不想开发完整 API

### D. 知识库/教程站

需求：

- 希望文章、教程、步骤、FAQ 可结构化输出
- 方便 Agent 做知识检索和问答

------

# 4. 核心问题

当前网站对 Agent 不友好的关键问题：

## 4.1 无统一入口

Agent 不知道先读什么、该信什么、哪里是核心信息。

## 4.2 无稳定结构

HTML 页面适合浏览，不适合高可靠抽取。

## 4.3 无内容语义层

网站没明确告诉 Agent：

- 这是什么页面
- 这段信息属于什么内容类型
- 哪个版本是最新的
- 哪个链接是主入口

## 4.4 无查询接口

网站有内容，但没有稳定、低歧义的对外查询方式。

## 4.5 无 Agent 开放控制

站长不能方便地定义：

- 哪些页面对 Agent 开放
- 返回摘要还是全文
- 哪些内容不该被机器读取

------

# 5. 产品方案

产品整体分成四层。

------

## 5.1 扫描层（Scan Layer）

作用：
 识别网站公开页面和内容类型。

能力：

- 读取网站首页、导航、sitemap
- 自动发现公开 URL
- 判断页面类型
- 建立基础页面清单

识别的页面类型包括：

- 首页
- 博客文章
- 文档页
- FAQ
- 产品页
- 定价页
- 关于页
- 联系页
- 更新日志页

------

## 5.2 语义层（Semantic Layer）

作用：
 给网页增加“Agent 可理解的语义分类”。

能力：

- 标记页面类型
- 提取标题、摘要、更新时间、标签、版本号
- 判断权威链接
- 标记主内容区域
- 建立页面间关系

输出目标：
 让 Agent 知道“这个网站怎么读”。

------

## 5.3 结构化层（Structured Output Layer）

作用：
 把网页信息输出为稳定的结构化数据。

核心输出包括：

- `agent-index.json`
- `agent-sitemap.json`
- `docs.json`
- `faq.json`
- `articles.json`
- `products.json`
- `pricing.json`
- `changelog.json`

这一层是产品的核心价值层。

------

## 5.4 查询层（Agent Query Layer）

作用：
 给 Agent 提供统一、稳定的公开查询方式。

可提供：

- 基础搜索
- 页面摘要查询
- FAQ 查询
- 产品信息查询
- 文档列表查询
- 文档版本查询

这层未来可以扩展为 CLI/MCP 输出，但 **MVP 不以 CLI 为主定位**。

------

# 6. MVP 定义

## 6.1 MVP 目标

先证明一件事：

**普通网站愿意为了“更容易被 AI 理解和引用”而接入这个工具。**

## 6.2 MVP 必须具备的能力

### 1. 网站扫描

- 输入网址即可扫描
- 支持 sitemap 导入
- 支持白名单/黑名单

### 2. 页面分类

- 能自动识别博客、文档、FAQ、产品等常见页面

### 3. Agent 入口文件生成

- 生成 `llms.txt` 或等价说明文件
- 生成 `agent-index.json`

### 4. 结构化导出

至少支持导出：

- articles.json
- faq.json
- docs.json

### 5. 基础查询能力

- search
- get_page
- get_faq
- list_docs

### 6. 开放规则配置

- 指定开放页面
- 指定开放类型
- 指定返回摘要/全文

------

## 6.3 MVP 不做

- 不做复杂可视化后台
- 不做企业级权限
- 不做完整统计系统
- 不做复杂多租户
- 不做本地软件控制接口
- 不做自动生成“软件 CLI”
- 不做高复杂度 Agent orchestration

------

# 7. 功能需求

## 7.1 模块一：网站接入

### 功能描述

用户输入网站 URL，系统自动扫描站点内容。

### 输入

- 网站 URL
- 可选 sitemap URL
- 可选内容规则文件

### 输出

- 站点页面清单
- 页面分类结果
- 可生成资源列表

### 验收标准

- 能发现主要公开页面
- 支持排除页面
- 支持重新扫描

------

## 7.2 模块二：Agent 入口生成

### 功能描述

为网站生成 Agent 首次访问的标准入口文件。

### 输出内容

- 网站简介
- 推荐读取顺序
- 核心资源清单
- 文档入口
- FAQ 入口
- 产品入口
- 更新时间
- 不建议读取内容说明

### 文件示例

- `llms.txt`
- `agent-index.json`
- `agent-sitemap.json`

### 验收标准

- 文件可自动生成
- 支持人工编辑
- 可重复构建

------

## 7.3 模块三：内容结构化导出

### 功能描述

将网站内容输出为稳定 JSON。

### 支持类型

- docs
- faq
- articles
- products
- pricing
- changelog

### 典型字段

#### docs.json

- id
- title
- url
- section
- version
- summary
- updated_at
- tags

#### faq.json

- id
- question
- answer
- category
- url
- updated_at

#### articles.json

- id
- title
- summary
- author
- published_at
- updated_at
- tags
- url

#### products.json

- id
- product_name
- description
- features
- url
- updated_at

### 验收标准

- 字段结构稳定
- 支持增量更新
- 支持字段映射
- 支持配置导出类型

------

## 7.4 模块四：基础查询 API

### 功能描述

对外提供只读查询接口。

### API 范围

- `search(query)`
- `get_page(id/url)`
- `get_faq(question/id)`
- `list_docs(section/version)`
- `list_articles(tag/date)`
- `get_product(name/id)`

### 返回要求

- JSON 格式
- 包含 source_url
- 包含 updated_at
- 支持摘要字段
- 支持分页

### 验收标准

- 响应结构稳定
- 查询延迟可接受
- 支持简单限流

------

## 7.5 模块五：Agent 开放规则

### 功能描述

允许站长控制给 Agent 看什么。

### 配置项

- 页面白名单/黑名单
- 内容类型开放控制
- 只开放摘要 / 开放全文
- 是否允许搜索
- 是否输出更新时间
- 是否允许导出产品/价格信息
- 访问频率限制

### 验收标准

- 配置文件可读易改
- 规则修改后可重新生成
- 最终生效规则可导出查看

------

## 7.6 模块六：更新检测

### 功能描述

内容变化后自动重建 Agent 可读层。

### 检测方式

- 手动重建
- 定时扫描
- hash 对比
- sitemap 对比

### 验收标准

- 可只更新变更页面
- 有更新时间记录
- 不因局部失败影响整体输出

------

# 8. 用户流程

## 8.1 站长流程

1. 输入网站 URL
2. 系统扫描公开页面
3. 系统识别页面类型
4. 用户配置开放范围
5. 系统生成 Agent 入口和结构化输出
6. 用户部署或托管
7. Agent 开始读取和查询

## 8.2 Agent 流程

1. Agent 发现网站 Agent 入口
2. 读取站点说明和资源索引
3. 根据任务选择数据源
4. 调用 JSON 或查询 API
5. 返回更准确结果

------

# 9. 差异化定位

这一部分是改版 PRD 最关键的。

## 9.1 不与“软件转 CLI”正面竞争

本产品不是：

- 把任意软件包装成 CLI
- 给本地软件增加控制命令
- 给 Agent 增加系统动作执行能力

本产品专注于：

- 网站公开内容
- 网站信息理解
- 网站数据结构化
- 网站对 Agent 的公开查询能力

## 9.2 核心价值主张

### 对站长

让你的网站更容易被 AI 正确理解、引用和推荐。

### 对文档站

让文档更容易被 Agent 检索，不容易读错。

### 对 SaaS 官网

让公开产品信息更容易被 AI 客服、销售 Agent 使用。

### 对博客/知识站

让内容更容易成为 AI 时代的输入源，而不是被忽略。

# 11. 技术形态建议

## 第一阶段：CLI 作为“生成工具”，不是“产品定位”

可以做 CLI，但只是为了方便站长生成文件，不是为了和“Agent CLI 平台”抢定位。

CLI 的职责：

- 扫描网站
- 生成输出
- 构建索引
- 发布到目标目录

## 第二阶段：托管服务

让非技术用户也能用。

## 第三阶段：插件化

进入真正适合被动收入的分发模式。

# 13. 验收标准

MVP 完成后，必须满足：

1. 输入网站 URL 可完成扫描
2. 可自动识别主要页面类型
3. 可生成 Agent 入口文件
4. 可导出至少 3 类结构化 JSON
5. 可按规则控制开放范围
6. 可执行增量更新
7. Agent 可通过标准 JSON / API 读取数据
8. 产品定位清晰为“Website-to-Agent Layer”，不混入软件控制层

你的真正定位应该是：

**让任何普通网站，在 AI 时代拥有被 Agent 稳定理解、检索、引用和查询的能力。**
